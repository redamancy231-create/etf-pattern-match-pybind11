# 為什麼 53× 最終變成 2.2×：pybind11 工作負載中的 Amdahl 定律

## 1. 引言

三個基準數字同時展示了這次 pybind11 重構的潛力與邊界。動態時間扭曲（DTW）距離的 C++ 實作比 Python 等效實作快 **34×**；一次完整的形態匹配——包括 15 維特徵擷取、DTW 精細排序和候選排序——達到 **53×**；但覆蓋 100 個時間點的批次工作負載只從 50 ms 降到 23 ms，端對端加速僅為 **2.2×**。

乍看之下，這組結果似乎互相矛盾。如果最具代表性的單次操作已經快了 53 倍，為什麼實際批次任務只快了兩倍多？換一種說法，當測量範圍從單次匹配擴大到完整工作流程時，為什麼加速比縮小了約 26 倍？

答案不是正確性缺陷、編譯器最佳化失敗，也不意味著 53× 是錯誤資料。不同數字測量的是不同邊界。微觀和中觀基準隔離了 C++ 能夠加速的程式碼；宏觀基準還包含 Python 控制流程、NumPy 與 C++ 之間的邊界處理、重複綁定分派，以及仍留在 Python 中的特徵。Amdahl 定律準確描述了這種現象：加速程式的一部分，無法消除其他部分所花的時間。因此，2.2× 的批次結果反而提供了更有價值的架構訊號。

## 2. 實驗設定

基準測試在同一臺機器上使用相同的合成輸入，對比 Python 與 C++ 實作，形態匹配演算法邏輯保持不變。處理器、記憶體系統、作業系統狀態和編譯器版本都會影響絕對耗時。由於成對結果來自同一硬體，相對加速比更具可比性，但它仍然只適用於當前工作負載，不能被視為普遍適用的 pybind11 常數。

| 項目 | 細節 |
|------|------|
| 硬體 | 消費級筆記型電腦——省略具體型號。就同一臺機器上的 Python 與 C++ 比較而言，相對加速比不取決於特定硬體；絕對耗時會隨硬體而異。 |
| Python | 3.12.7 + NumPy |
| C++ | C++20、MSVC 19.51（Visual Studio 2026 Community）、pybind11 3.0.4 |
| 編譯選項 | <code>/O2 /arch:AVX2</code>，Release 組態 |
| 測試資料 | 20 檔 ETF × 1005 個交易日，合成隨機漫步價格序列（<code>seed=42</code>） |
| 測量方式 | <code>time.perf_counter()</code>，3 輪取最小值，每輪 100 次迭代 |
| 可重現命令 | <code>python verify_etf_core.py</code> + <code>python verify_batch.py</code> |

這裡使用合成資料是合理的，因為目標是測量執行成本，而不是模擬真實市場。固定隨機種子使工作負載可重複，一致性腳本則檢查 C++ 實作是否在專案規定的浮點容差內保持 Python 演算法輸出。因此，這些數字應被理解為兩種實作的工程比較，而不是回測，更不是投資表現證據。

## 3. 三層基準結果

| 函式 | Python | C++ | 加速比 |
|----------|--------|-----|:---:|
| DTW 距離（L=19） | 96 µs | 2.8 µs | **34×** |
| 單次形態匹配 | 14.0 ms | 0.26 ms | **53×** |
| 批次匹配（×100） | 50 ms¹ | 23 ms | **2.2×¹** |

> ¹ 批次資料列比較的是 100 次 C++ 單次呼叫與 1 次 C++ 批次呼叫——衡量批次介面所降低的開銷，非 Python vs C++ 加速比。50 ms 基線為 100 × C++ `pattern_match_single`，非 100 × Python。

### 3.1 第一層——微觀：單一運算核心

DTW 基準是在長度為 19 的輸入上，比較一個 C++ 函式與其 Python 等效實作。測量邊界刻意保持狹窄：主要成本來自巢狀數值迴圈，外圍編排幾乎不計入計時區間。陣列跨過綁定邊界後，C++ 使用編譯後的迴圈和連續數值儲存執行動態規劃遞推，計時範圍內幾乎不再有 Python 工作。

因此，當直譯器層級的迴圈被最佳化後的原生程式碼取代時，**34×** 是合理結果。它是一項有效測量，但回答的問題很窄：在當前輸入維度下，此核心單次呼叫能快多少？它不能回答整個應用程式最終會快多少。

### 3.2 第二層——中觀：一次完整形態匹配

單次匹配基準擴大了邊界，其中包括 15 維特徵生成、候選過濾、DTW 評分和排序。完成首次呼叫後，各階段繼續在 C++ 內部串聯，中間值不會反覆返回 Python。原生實作還可以讓熱點迴圈、暫存向量和排序邏輯保持在同一個編譯後的執行區段中。

結果是 **53×**，甚至高於獨立 DTW 的加速比。這是合理的，因為 C++ 路徑加速的不只是 DTW：它還消除了多個協同迴圈中的 Python 開銷，並避免在內部階段之間建立 Python 物件。這裡的結論並不是 pybind11 自動讓函式快 53 倍，而是：把足夠大、運算足夠密集的工作單元放在一次綁定呼叫之後，才能獲得這種收益。

### 3.3 第三層——宏觀：100 個時間點的批次任務

批次基準量測的是工作流程，而不是獨立核心。處理 100 個時間點時，單次原生呼叫中幾乎看不見的成本會顯現出來：基線路徑中的 Python 迴圈分派、NumPy 陣列準備、參數驗證、緩衝區處理、綁定邊界切換、結果建構，以及僅在 Python 中執行的特徵運算。C++ 批次 API 減少了跨界次數，並重複使用了高度重疊的候選視窗計算，但它無法清除加速區域之外的全部工作。

因此，耗時從 50 ms 降到 23 ms，得到有用但並不誇張的 **2.2×** 加速。這三個數字不是相互競爭的結論，而是一個層級：34× 描述運算核心，53× 描述原生運算單元，2.2× 描述實際觀察到的批次架構。基準邊界越寬，其中包含的未加速工作就越多。

## 4. Amdahl 定律分析

### 4.1 第一步：建立模型

Amdahl 定律將工作負載分為可加速部分和不受本次最佳化影響的部分：

~~~text
S = 1 / ((1 - p) + p/s)

其中：
  S = 觀察到的端對端加速比（2.2）
  s = 被加速部分的加速比（53）
  p = 總工作量中可以被加速的比例
~~~

這是一個簡化模型。它假設可加速部分按固定倍數縮短，剩餘部分保持不變。實際的 pybind11 程式還可能增加或減少記憶體配置、快取、轉換和排程成本，所以運算結果應視為架構估計，而不是效能分析器的直接輸出。

### 4.2 第二步：求解可加速比例

整理公式可得：

~~~text
已知 S = 2.2，s = 53：
p = (S - 1) × s / ((s - 1) × S)
  = (1.2 × 53) / (52 × 2.2)
  ≈ 0.556
~~~

在該模型下，原始批次工作負載中只有約 **56%** 屬於能獲得 53× 原生加速的部分；另外 **44%** 相對於本次最佳化仍是循序的，包括 Python 端編排、語言邊界處理，或尚未遷移到 C++ 的運算。

必須注意，這是相對於基線的解釋。若把原始 50 ms 視為 100%，約 28 ms 可加速，22 ms 不可加速。將 28 ms 縮短 53 倍後只剩約 0.53 ms（≈0.5 ms 四捨五入），總耗時預測值約為 22.5 ms，與觀察到的 23 ms 非常接近。這正是獨立核心的巨大加速與有限的端對端結果能夠同時成立的原因。

### 4.3 第三步：分解 44% 的開銷

44% 是 Amdahl 等效估計，而不是效能分析器直接測出的明細。對目前的批次架構，可以採用如下實用分解：

| 開銷來源 | 估算比例 | 機制 |
|-----------------|:------:|-----------|
| Python for 迴圈分派 | ~15% | Python 編排路徑中 100 次迭代的直譯器開銷 |
| NumPy ↔ C++ 資料轉換 | ~15% | <code>py::array</code> ↔ <code>numpy.ndarray</code> 的緩衝區處理，以及所有權、dtype 或資料排列方式不符合條件時產生的複製 |
| pybind11 呼叫開銷 | ~8% | 每次跨語言呼叫的參數解析、驗證、封裝分派和 GIL 管理 |
| Python 端特徵運算 | ~6% | 尚未遷移到 C++ 的 F16, F17, F20, F21 市場環境特徵 |

這些估算合計佔基線工作負載的 44%。它們用於找出最佳化方向，而不是宣稱亞毫秒級測量精度。特別需要說明：當 dtype、連續性和生命週期條件滿足時，NumPy 緩衝區協定可以實現零複製；這裡的類別還包括驗證、建立檢視物件、強制型別轉換、結果緩衝區配置以及無法避免的複製。

### 4.4 第四步：達到 10× 需要什麼？

假設 C++ 核心變得無限快。令 <code>s → ∞</code>，可加速項消失，但循序比例仍然存在：

~~~text
S_max = 1 / (1 - p)
      = 1 / (1 - 0.555…)
      ≈ 2.25×
~~~

觀察到的 **2.2×** 已經非常接近目前架構隱含的約 **2.25×** 上限（若用四捨五入後的 p=0.56→0.44 拆分，會得到 2.3×，但精確 p≈0.555… 給出的是 2.25×；差異很小，不影響結論）。繼續調整 DTW 或排序邏輯或許還能縮短剩餘約 0.48 ms 的可加速部分，但只要基線中仍有 44% 位於該區域之外，就不可能得到 10× 的端對端效能。

若要達到 10×，即使假設核心無限快，循序比例也必須低於 10%。在核心加速比有限、即 53× 的情況下，將 <code>S = 10</code> 代入公式可得 <code>p ≈ 0.917</code>：約 92% 的工作負載必須能夠被加速。這意味著需要將更多工作流程——而不僅僅是更快的算術——納入 C++ 邊界內，包括時間點迴圈、可合理移轉的資料準備、F16, F17, F20, F21 運算、中間聚合和最終結果封裝。因此，接下來最佳化的重點是邊界設計。

## 5. 工程啟示

### 5.1 pybind11 適合運算密集、呼叫頻率低的工作負載

一次跨語言呼叫存在固定成本。社群基準測試和 pybind11 文件顯示，每次呼叫約 0.5–2 µs，具體取決於函式簽章複雜度、參數轉換、平台和建置組態（這是一般數量級估計，非本專案實測值）。對於一個原本耗時 15 ms、遷移後仍形成較大原生任務的操作，這點成本可以忽略；對於本身只需 2.9 µs 的 DTW 呼叫，它就明顯得多。如果從 Python 呼叫這類小型函式數千次，即使 C++ 函式主體非常優秀，進入和離開 C++ 也會佔據可觀時間。解決方法是公開粒度較粗的操作。

### 5.2 批次介面是效益最高的最佳化

將介面從 Python 編排的 100 次獨立呼叫改為 1 次批次呼叫，產生了測得的 **2.2×** 提升。批次函式只需驗證一次共用輸入、跨越一次語言邊界、重複使用重疊的候選視窗，並集中建構輸出。這類架構效益往往比從已經最佳化的內部迴圈中再擠出幾個百分點更有價值。良好設計的 pybind11 API 應減少跨界頻率，而不只是降低單次跨界成本。

這也指出了下一步方向：繼續擴大批次邊界。如果 Python 仍在計算 F16, F17, F20, F21、重複建立檢視物件，或逐項處理結果，就應評估是否能將這些階段合併進單次端對端原生管線，同時保證 API 仍然可測試、可維護。

### 5.3 加速比必須說明適用情境

**53×** 是準確的微觀/中觀基準，也是很醒目的 README 摘要；**2.2×** 則更能代表測得的批次工作負載。兩者都是真的，因為它們回答不同問題。只報告較大的數字會隱藏系統層級瓶頸；只報告較小的數字又會掩蓋所抽取出的原生核心之實際成效。

因此，效能報告應明確基準邊界、輸入規模、呼叫次數、測量方法和仍留在 Python 中的工作。一份有用的報告會同時給出核心與端對端結果，並解釋二者之間的差距。在本專案中，這個差距不是需要迴避的問題，而是最重要的工程發現。
